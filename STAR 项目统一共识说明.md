# STAR 项目统一共识说明（内部基线稿 / v2）

> 用途说明：
> 本文不是面向教师、评审或最终读者的正式文档，而是作为后续讨论、分析、设计与写作时的内部稳定事实源。后续若无新的明确决议，默认以本文为准，避免因参考初稿频繁变动而引发歧义。

------

## 1. 文档定位

STAR 当前被明确定位为一个 **CS 模式的自学课程社区与学习进度管理系统**，而不是以单机离线桌面应用为核心定位的个人工具。

系统的核心不是“存放资料”，也不是“简单待办清单”，而是围绕“课程”这一数据单元组织：

- 课程是某位作者组织的一套学习方案、学习路线或自学计划。
- 课程 **不是** 对现实学科的一一映射。
- 同一学科下可以存在多门不同课程，它们拥有不同作者、不同结构、不同风格与不同编号。
- 系统既服务于个人自学，也服务于社区共享、参考、派生与改造。

这一社区设计理念受到 GitHub 开源社区与 CSDIY 一类计算机自学路线社区的启发：

- 一门课程可以类比为一个“仓库式学习路线”；
- 社区中的用户可以检索、参考、学习、fork、改造并再发布课程；
- 系统强调课程内容本体与用户个人学习状态之间的解耦。

------

## 2. 当前已经明确废弃的旧前提

以下内容属于旧版设计稿中的历史性假设，当前已不再作为系统真实前提使用：

1. 系统是 Windows 单机桌面应用。
2. 系统不依赖服务器，仅本地运行。
3. SQLite 本地数据库是系统的核心唯一存储方案。
4. 公开课程仅作展示，缺少完整社区传播语义。
5. 课程编辑逻辑默认围绕“本地个人对象”展开，而非围绕“社区课程本体 + 发布版本 + 学习关系”展开。

这些旧假设在后续架构设计、数据库设计、接口设计与模块说明中都不应被继续默认沿用。

------

## 3. 系统总目标

STAR 的目标可以概括为：

### 3.1 课程化组织自学内容

系统以课程为核心组织学习内容，帮助作者将零散资源、章节安排和知识点规划组织成结构化自学路线。

### 3.2 支撑个人学习进度管理

用户可以将课程纳入自己的学习计划，并对课程内的知识点与资源进行学习状态记录，系统据此计算章节进度与课程进度。

### 3.3 支撑社区共享与参考

用户可将课程发布到社区中供其他用户浏览、检索、参考与学习。

### 3.4 支撑 fork 式传播与派生改造

用户可基于他人的课程创建派生课程，在保留来源关系的前提下形成自己的独立课程路线。

### 3.5 支撑作者持续维护与版本化发布

作者可以持续维护课程草稿，并在合适的时候发布新的公开版本；学习者则学习某个稳定发布版本，而不是被作者的即时修改持续打断。

------

## 4. 课程的基本语义

### 4.1 什么是课程

课程表示“作者组织的一套学习方案”。

它至少具有以下特点：

- 有自己的课程编号；
- 有作者；
- 有标题、简介、标签等元信息；
- 有明确的内部结构；
- 可以处于私有草稿态，也可以拥有对社区公开的发布版本；
- 可以被其他用户学习，也可以被其他用户 fork。

### 4.2 课程不是学科唯一实体

例如：

- “操作系统自学路线 A”
- “操作系统八股速成路线”
- “MIT 风格操作系统系统学习路线”

都可能属于“操作系统”这一学科，但在本系统中应视为不同课程对象。

因此，系统不以“学科唯一性”组织课程，而以“课程作者 + 课程结构 + 课程意图”组织课程。

------

## 5. 课程来源

课程的来源已经明确为三类：

### 5.1 官方预置课程

由系统设计者 / 平台侧预置的一批课程。

### 5.2 用户手工创建课程

用户可从空白开始，自主创建课程并填写其结构与资源。

### 5.3 AI 辅助生成课程

用户可通过 AI 接口生成课程初稿，再由用户自行修改、增删与完善。

因此，AI 在这里的定位是“辅助课程生成工具”，而不是课程唯一来源，也不是替代用户编辑的黑箱自动生成器。

------

## 6. 课程内部结构（已冻结）

课程内部采用如下结构：

**课程 → 章节 →（知识点 + 资源）**

这是当前已经冻结的最终方案。

### 6.1 章节

章节是课程的线性组织单元。

- 一门课程由若干章节按顺序构成；
- 章节主要承担结构组织作用；
- 章节不是最细进度粒度。

### 6.2 知识点

知识点是课程中的 **主学习单元**，表示“这一小块内容是否学会 / 学到何种程度”。

知识点的语义是：

- 表示要掌握的学习目标；
- 是主进度统计的核心来源；
- 是学习状态记录的主要粒度。

### 6.3 资源

资源是课程中的 **辅助学习材料**，表示“通过什么材料去学习”。

资源的语义是：

- 资源不是学习目标本身；
- 资源服务于知识点的学习；
- 资源本身可以记录使用状态，但不应成为主进度计算的核心粒度。

### 6.4 知识点与资源的关系

当前已确定如下规则：

- 资源先隶属于某个章节；
- 资源可选地关联一个或多个知识点；
- 因此资源既可以是章节总览型资源，也可以是单知识点资源，也可以是多知识点综合资源。

这意味着资源关系不是死板的一对一，而是允许灵活表达真实学习材料的覆盖范围。

------

## 7. 进度语义

### 7.1 进度不属于课程本体

进度不是课程对象的固有属性，而是 **用户对课程的学习状态**。

因此，进度必须绑定在：

**用户-课程学习关系**

上，而不能直接绑定在课程本体中。

### 7.2 用户不拥有课程本体，只拥有与课程的学习关系

更准确地说：

- 用户可以是课程作者；
- 但用户对课程的“学习状态”与“课程内容本体”是两回事；
- 作者本人也可以对自己创建的课程建立学习关系，并记录进度。

### 7.3 主进度粒度

主进度以 **知识点级** 为核心。

推荐语义：

- 章节进度 = 本章已完成知识点数 / 本章知识点总数
- 课程进度 = 全课程已完成知识点数 / 全课程知识点总数

### 7.4 资源状态

资源可以记录辅助状态，但它不构成主进度的主要依据。

资源状态更适合表达：

- 是否已查看
- 是否已完成
- 是否不采用
- 是否待补充

### 7.5 这样设计的原因

采用“知识点为主、资源为辅”的进度语义，主要因为：

1. 资源数量不稳定；
2. 一个知识点可能对应多个资源；
3. 一个资源也可能覆盖多个知识点；
4. 若把资源作为主进度粒度，会造成统计偏差与迁移困难。

------

## 8. 两个关键动作：加入学习 与 fork（已冻结）

### 8.1 必须拆分为两个动作

当前已经正式确定：

**加入学习** 与 **fork** 是两个语义完全不同的动作，不可混用。

### 8.2 加入学习

加入学习表示：

- 用户决定学习某门课程；
- 系统创建一条“用户-课程学习关系”；
- 用户后续在这条关系上记录自己的进度；
- 加入学习不会复制课程本体。

从产品展示角度，未来 UI 中可以考虑使用更有产品感的表达，例如：

- 加入计划
- 开始学习
- 纳入计划

但在系统设计语义中，仍建议用“加入学习 / 建立学习关系”作为稳定术语。

### 8.3 fork

fork 表示：

- 用户基于某门课程的某个公开发布版本，复制出一门新的独立课程；
- 新课程立刻生成新的课程编号；
- 数据库中创建新的课程本体条目；
- 新课程与原课程解绑，仅保留来源关系；
- fork 后的课程由 fork 作者自己维护；
- fork 后课程的公有 / 私有由用户自行设置。

因此，fork 的本质是：

**复制并改造，形成新的独立课程路线。**

它不是“加入学习”，也不是“收藏”，也不是“引用”。

### 8.4 fork 关系的意义

fork 需要保留来源信息，后续至少应支持：

- 标明来源课程；
- 显示 fork 次数；
- 支持社区理解课程传播关系；
- 在需要时支持展示派生链条。

------

## 9. 课程内容本体 与 用户学习状态 的分离（核心原则）

这是后续所有设计中必须反复坚持的核心原则：

### 9.1 课程内容本体

课程内容本体包括：

- 课程标题、简介、标签
- 章节结构
- 知识点定义
- 资源定义
- 发布版本
- fork 来源关系

### 9.2 用户学习状态

用户学习状态包括：

- 是否加入学习
- 当前学习绑定的版本
- 知识点完成状态
- 资源使用状态
- 学习记录
- 学习统计信息

### 9.3 二者必须解耦

不得把：

- 章节状态
  n- 课程完成状态
- 资源完成状态

作为课程本体的固有属性直接写入课程结构中，否则会破坏社区共享语义与多用户学习语义。

------

## 10. 课程的“仓库态”与“发布态”模型（已冻结）

为了既支持作者持续编辑，又保护学习者使用稳定内容，当前已经确定采用“双层对象”思路：

- **课程本体 / 课程仓库 / 工作区**：持续存在、由作者维护的连续对象
- **课程发布版本**：从课程当前草稿中发布出来的稳定快照

### 10.1 课程本体

课程本体表示一条持续存在的课程路线身份。

它更像仓库、工作区或项目本体，具有以下特征：

- 有课程编号（course_id）；
- 属于某位作者；
- 可持续编辑；
- fork 会生成新的课程本体；
- 是“这门课”的身份载体；
- 可拥有草稿态与一个或多个发布版本。

### 10.2 私有课程的本质

私有课程主要表现为课程本体的 **草稿态**：

- 仅作者自己可见和可编辑；
- 尚未发布到社区；
- 可频繁修改；
- 产品语义上不需要强调正式版本概念。

私有阶段可以存在内部修订，但不必在产品层暴露复杂版本语义。

### 10.3 发布到社区

当作者将课程发布到社区时：

- 不生成新的课程本体；
- 而是从当前草稿生成一个新的 **公开发布版本**；
- 社区中的用户学习与浏览的是该公开发布版本；
- 课程本体仍由作者继续维护。

### 10.4 课程发布版本

发布版本表示某一时刻冻结出来的稳定快照，特点是：

- 应避免被直接原地修改；
- 应可供学习者稳定依赖；
- 同一课程本体可拥有多个发布版本；
- 不同版本之间形成连续演进关系。

### 10.5 学习者绑定发布版本

用户加入学习时，默认绑定该课程的 **最新公开版本**。

后续其学习关系可以固定在特定版本上，不因作者编辑草稿而自动漂移。

### 10.6 版本更新

作者后续若修改课程：

- 修改首先发生在课程草稿上；
- 再次发布时生成新版本；
- 学习旧版本的用户不会被强制打断；
- 系统可以提示用户是否迁移到新版本。

### 10.7 为什么不把每次公开更新都视为 fork 新课程

因为作者对同一路线的维护更适合表达为“新版本”，而非“新的派生课程”。

若把正常维护也视为 fork，会造成：

- 课程编号爆炸；
- 搜索结果被版本刷屏；
- fork 语义被稀释；
- 同一门课看起来像很多门重复课程；
- 派生树被普通版本更新污染。

因此已经明确：

- **fork 用于独立派生新课程本体**
- **version / release 用于同一课程本体的迭代发布**

------

## 11. 公有课程编辑语义（已冻结）

### 11.1 私有课程

私有课程可原地自由编辑，仍视为同一课程对象的连续草稿状态。

### 11.2 公有课程

公有课程的学习者依赖的是其某个公开发布版本，因此：

- 不应让作者的后续编辑直接覆盖学习者正在学习的稳定版本；
- 作者对课程的后续改动应先进入草稿；
- 再通过重新发布形成新版本。

### 11.3 结构性修改与非结构性修改

在更细的实现层面，可以进一步区分：

- 非结构性修改：标题润色、简介修正、标签调整、说明文字修改等；
- 结构性修改：章节增删、顺序调整、知识点重组、关键资源替换等。

最终实现上可进一步讨论哪些改动允许直接作用于当前版本的公开元信息，哪些必须通过新版本发布。但总体原则已经确定：

**不能直接破坏其他学习者所依赖的公开学习版本。**

------

## 12. 课程版本与 fork 的区别（必须固定）

### 12.1 版本（Version / Release）

版本表示：

- 同一课程本体的迭代发布；
- 由原作者维护；
- 表示同一路线的连续演化；
- 不生成新的课程本体编号。

### 12.2 fork

fork 表示：

- 基于某课程的某个发布版本创建新的独立课程本体；
- 生成新的课程编号；
- 保留来源关系；
- 后续走自己的发展线；
- 公有 / 私有由新作者自行决定。

因此，两者不可混淆。

------

## 13. 社区可见性与检索（当前已明确部分）

### 13.1 公有课程

公有课程可被社区中的其他用户：

- 检索
- 访问
- 参考
- 加入学习
- fork

### 13.2 私有课程

私有课程默认仅用户本人使用。

是否支持邀请 / 分享他人加入私有课程，当前仍属于 **未决策项**，后续可再讨论。

### 13.3 社区检索的一期最小字段

当前已确定：一期检索最小字段可以包括：

- 课程名
- 课程编号（课程本体编号）
- 标签
- 作者
- 简介关键词

### 13.4 社区辅助参考信息

课程可拥有或后续扩展拥有如下社区信息：

- fork 次数
- 评论
- 其他参考指标

其中评论功能存在价值，但是否作为一期主功能，仍可按实现复杂度择机收敛。

------

## 14. 作者与学习者的关系

当前已明确：

- 作者是课程本体的拥有者与维护者；
- 作者本人也可以对自己创建的课程建立学习关系；
- 即“拥有课程”与“学习课程”是两个不同维度；
- 同一用户可以同时是某课程的作者和该课程的学习者。

------

## 15. 学习关系的推荐语义

虽然最终数据库设计尚未展开到表级，但从概念上已经确定：

用户加入学习时，会形成一条 **用户-课程学习关系**，至少承担以下职责：

- 指明是谁在学哪门课程；
- 记录当前绑定的发布版本；
- 承载知识点级进度；
- 承载资源使用状态；
- 支撑章节 / 课程进度统计；
- 支撑“有新版本可升级”的提示。

因此，后续数据库设计中，学习关系应作为重要的一层对象，而不是被弱化成零散学习记录的附属表。

------

## 16. 进度迁移的基本态度

当课程存在新版本时，旧版本学习者可以选择是否迁移。

当前已经明确的态度是：

- 允许用户选择是否升级到新版本；
- 不强制自动升级；
- 一期不要承诺复杂的“智能完美进度转换”；
- 更适合采用基于稳定节点标识的有限迁移策略。

也就是说：

- 能映射的知识点状态则迁移；
- 无法可靠映射的部分则要求用户重新确认。

这比一开始就设计重量级自动转换机制更稳妥。

------

## 17. 资源的一期范围（已冻结）

一期明确不做复杂文件托管系统。

### 17.1 当前范围

资源以 **链接 + 元数据** 为主，例如：

- 网课链接
- 电子书链接
- 文档链接
- 论文链接
- 下载地址
- 其他在线学习材料入口

### 17.2 资源应具备的基本说明

资源除了链接本身，还应能说明：

- 标题 / 名称
- 类型
- 涵盖内容
- 所属章节
- 可选关联知识点

### 17.3 后续扩展

若未来支持本地托管文件或其他文件资源形态，可视作后续扩展，而非一期核心范围。

------

## 18. 账号体系（已冻结）

系统必须具备账号登录体系。

原因在于：

- 课程本体与作者绑定；
- 用户学习关系依赖用户身份；
- fork 行为依赖明确的操作者；
- 公有 / 私有控制依赖权限体系；
- 社区检索、发布、学习记录等都需要账号上下文支撑。

因此，账号体系不是附属功能，而是系统基本前提之一。

------

## 19. 一期最值得保护的核心主链路

从产品最小闭环看，一期最重要的是打通以下链路：

1. 用户登录
2. 创建 / AI 生成课程草稿
3. 编辑课程结构（章节、知识点、资源）
4. 发布课程到社区
5. 社区检索课程
6. 用户加入学习
7. 用户记录知识点与资源状态
8. 系统计算学习进度
9. 用户 fork 公有课程形成自己的独立课程

只要这个主链路通顺，系统就已经具有鲜明辨识度。

相对而言，以下内容可以作为后续增强项审慎扩展：

- 评论系统的复杂互动能力
- 高级推荐
- 复杂进度迁移
- 本地文件托管
- 协作编辑与私有邀请
- 更复杂的版本对比能力

## 20. 当前仍保留为未决策项的内容

这些未决策项不影响当前总架构、对象模型与主语义的稳定性。

## 21. 后续讨论时必须默认遵守的术语

为避免后续歧义，建议默认使用以下术语：

- **课程本体 / 课程仓库 / 工作区**：持续存在、可编辑的课程对象
- **发布版本**：课程发布到社区时形成的稳定快照
- **加入学习**：建立用户-课程学习关系
- **fork**：复制并改造课程，生成新的独立课程本体
- **知识点**：主学习单元
- **资源**：辅助学习材料
- **学习关系**：用户与课程之间用于追踪进度的关系对象

并默认避免以下混淆：

- 不把 fork 说成加入学习；
- 不把版本说成 fork；
- 不把课程进度写成课程本体属性；
- 不把资源误当作主学习目标；
- 不把课程视为学科唯一实体。

## 22. 当前最终总结

截至当前对话，STAR 的稳定共识可以浓缩为以下一句话：

**STAR 是一个以课程为核心组织单元的 CS 模式自学课程社区与学习进度管理系统。课程是作者组织的一套学习路线，而非现实学科的一一映射；课程内部采用“章节—知识点—资源”的组织思想，其中知识点是主学习单元、资源是辅助材料。系统区分课程内容本体与用户学习状态，区分加入学习与 fork，区分课程仓库草稿态与公开发布版本。作者维护课程草稿并发布版本，学习者绑定稳定版本进行学习，其他用户可基于公开版本 fork 出新的独立课程。**

若后续无新的明确决议，默认以上全部内容为后续系统设计、数据库建模、模块划分、接口设计和说明文档撰写时的最高优先级事实依据。
